# 统一审批卡体验设计

## 背景

当前会话流里的审批请求虽然已经接通了协议层能力，但前端呈现仍停留在“底层 request 调试卡”阶段：

- 标题直接暴露 `item/commandExecution/requestApproval` 这类 method 名
- 命令审批没有完整展示命令、执行目录、触发原因、后续授权建议
- 文件变更审批和命令审批没有统一的视觉与交互模型
- 用户只能看到一排技术按钮，感知上更像调试面板，而不是可理解、可决策的产品化审批

从现有协议与类型定义看，命令审批支持：

- `accept`
- `acceptForSession`
- `acceptWithExecpolicyAmendment`
- `decline`
- `cancel`

因此问题不是“协议没有提权/授权能力”，而是 UI 没有把已有能力和上下文信息转译成用户能理解的体验。

## 目标

- 将命令审批与文件变更审批统一升级为同一套中文产品化体验
- 保持审批仍然发生在会话流内，贴近 Codex App 的使用感受
- 让用户在审批前先看到足够的信息，再做选择
- 对命令审批补齐“本次允许 / 会话内减少重复确认 / 规则授权同类命令”的展示能力
- 保留现有非审批类 server request 的回退渲染，不在本次范围内整体重做
- 保证桌面端与移动端都可用

## 非目标

- 不改动 app-server 协议、schema 或 `documentation/app-server-schemas/`
- 不将审批改成全局模态弹窗或脱离会话流的抽屉
- 不在第一版重做 `item/tool/requestUserInput`、`item/tool/call` 等其他 request 类型
- 不新增复杂的审批历史页或审批中心
- 不在第一版增加 toast、系统通知或声音提醒

## 推荐方案

### 1. 交互形态

采用“会话内统一审批卡 + 准弹层视觉”：

- 卡片仍嵌在 `ThreadConversation.vue` 的会话流里
- 宽度、阴影、圆角、分区留白明显高于普通消息气泡
- 标题、正文、选项、操作区分层展示
- 真正提交前，先通过单选式选项选择审批决定，再点击底部提交按钮确认

这样既保留上下文连续性，也能做出接近截图里的强聚焦审批体验。

### 2. 统一壳子，分类型内容

审批卡分为两层：

- 统一审批壳子：负责标题、说明、选项区、底部提交区、卡片骨架
- 类型化内容区：根据 request 类型渲染命令审批内容或文件变更审批内容

第一版覆盖：

- `item/commandExecution/requestApproval`
- `item/fileChange/requestApproval`

其他 request 继续走现有降级渲染逻辑。

### 3. 命令审批卡信息结构

命令审批卡建议分为 5 个区块：

#### 标题区

- 标题：`是否允许执行此命令？`
- 副标题：`该操作需要你的授权后才能继续`

如果请求里带有更强的原因信号，可根据 `reason` 文案显示更具体的说明，例如：

- `该操作可能需要突破当前沙箱限制`
- `该操作需要额外网络访问权限`

#### 命令区

- 用等宽命令块完整展示 `command`
- 在命令块下补充：
  - `执行目录`
  - `触发原因`

当协议里存在 `commandActions` 时，额外转译出简短摘要，例如：

- `预计操作：读取目录、搜索文件`

#### 影响说明区

当请求里包含 `proposedExecpolicyAmendment` 时，展示业务化说明：

- `你也可以授权同类命令在后续自动通过，减少重复确认`

不直接暴露底层字段名，但允许在展开说明或次级文字中展示规则摘要文本。

#### 选项区

做成编号式可选项，默认不自动提交：

- `1. 允许本次执行`
- `2. 允许本次执行，并对后续同类命令减少重复确认`
- `3. 拒绝，并让 Codex 调整方案`

映射规则：

- 有 `proposedExecpolicyAmendment` 时，第 2 项提交 `acceptWithExecpolicyAmendment`
- 没有该字段时，第 2 项退化为 `acceptForSession`
- 第 3 项提交 `decline`

保留 `cancel` 作为非主路径操作，不放进主选项组，避免干扰主要决策。

#### 操作区

- 主按钮：`提交`
- 次按钮：`跳过`

只有用户明确提交时，才真正向后端回复审批结果。

### 4. 文件变更审批卡信息结构

文件变更审批卡沿用同一壳子，但替换内容区：

#### 标题区

- 标题：`是否允许应用这些文件改动？`
- 副标题：`这些改动需要你的确认后才能写入工作区`

#### 变更摘要区

- 展示文件数统计
- 展示总增删行统计
- 展示前若干个文件路径摘要
- 保留“查看完整 Diff”入口，继续复用现有 diff 能力

#### 选项区

- `1. 允许应用本次改动`
- `2. 允许本次改动，并在本会话内减少同类确认`
- `3. 拒绝这次改动`

映射规则：

- 第 1 项：`accept`
- 第 2 项：`acceptForSession`
- 第 3 项：`decline`

同样保留 `cancel` 作为次级路径。

### 5. 视觉风格

整体视觉方向为“中文产品化、准弹层、强聚焦”：

- 背景使用高于普通消息的提升层
- 边框和阴影显著，但不做全屏遮罩
- 标题区、内容区、选项区之间有明确的节奏分隔
- 命令块和文件摘要块使用次一级面板承载
- 选项采用整行可点击卡片或单选项，避免继续使用纯按钮排布

文案层面避免暴露协议术语：

- 不直接显示 `acceptForSession`
- 不直接显示 `requestApproval`
- 不把 method 名作为标题

### 6. 响应式策略

桌面端：

- 审批卡居中显示，最大宽度高于普通消息
- 选项区纵向排列
- 底部按钮右对齐或两端分布

移动端：

- 卡片占满可用宽度
- 命令块允许横向滚动
- 文件列表默认摘要化，避免超长列表撑爆布局
- 提交和跳过按钮改为纵向或等宽堆叠，降低误触

## 备选方案对比

### 方案 A：仅在现有卡片上补字段和按钮

优点：

- 改动最小
- 风险最低

缺点：

- 只能解决“信息不全”
- 很难做出接近 Codex App 的产品化层级

### 方案 B：会话内统一审批卡

优点：

- 最贴近当前目标和 Codex App 感受
- 可以统一命令审批与文件审批的交互语言
- 改动集中在前端展示层，不涉及协议改造

缺点：

- 需要新增组件和展示 helper
- 需要补一轮样式与回归验证

### 方案 C：真正改为全局弹窗

优点：

- 打断感最强

缺点：

- 与 Codex App 的会话流体验不够贴近
- 焦点管理、移动端和多请求并发处理复杂度更高

结论：第一版采用方案 B。

## 实现落点

- `src/components/content/ThreadConversation.vue`
  - 将审批请求挂载到统一审批卡组件
  - 保留其他 request 的降级渲染
- `src/components/content/`
  - 新增统一审批卡组件
  - 新增命令审批内容组件或同文件内的类型化分支
  - 新增文件变更审批内容组件或同文件内的类型化分支
- `src/utils/`
  - 新增审批请求展示层解析 helper
  - 负责把 request `params` 转成 UI 可消费的数据结构
- `src/i18n/uiText.ts`
  - 增加中文产品化文案，同时补英文占位文案
- `tests/`
  - 增加审批展示与文案回归测试

## 测试建议

### 自动化

- 命令审批请求能解析出命令、目录、原因、建议授权选项
- 文件变更审批请求能解析出展示摘要与审批选项
- 当存在 `proposedExecpolicyAmendment` 时，命令审批第二选项映射为规则授权
- 当不存在该字段时，命令审批第二选项回退为会话内授权
- `ThreadConversation.vue` 已接入统一审批卡，而不是继续渲染 method 名

### 手工验证

- 命令审批卡能清楚展示命令、目录、原因
- 文件变更审批卡能展示统计和文件摘要，并能跳转完整 diff
- 桌面端和移动端都能完成选择与提交
- 深色 / 浅色主题下卡片层级清晰
- 非审批 request 仍能正常回退显示

## 结论

第一版最合适的落地方式是：

- 保持审批在会话流中
- 抽统一审批卡壳子
- 为命令审批和文件变更审批分别提供产品化内容区
- 使用中文产品化文案替换底层协议术语
- 通过单选式选项 + 提交按钮完成审批，而不是直接暴露一排技术按钮

这样既能补齐当前“看起来没有提权能力”的核心缺口，也能在不改协议的前提下，把现有审批能力真正变成用户可理解、可操作的产品体验。
