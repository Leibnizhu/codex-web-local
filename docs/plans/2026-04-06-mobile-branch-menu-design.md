# 移动端分支列表展示优化设计

## 背景

当前分支入口位于输入器底部状态区中部，桌面端点击后通过相对触发器的浮层菜单展示分支列表、阻塞原因与新建分支表单。

在手机窄屏下，现有菜单仍使用固定宽度与 `right-0` 的绝对定位展开：

- 分支菜单宽度固定为 `w-64`
- 菜单相对分支 chip 右对齐向左展开
- 分支入口本身并不贴近屏幕最右侧

因此移动端打开菜单时，整个面板会明显向左越过视口，被屏幕左边缘裁切，造成：

- 分支名首部被遮挡
- 标题、提示与表单不完整
- 长分支列表难以滚动和点击

这不是数据问题，而是移动端交互形态与定位策略不匹配。

## 目标

- 修复手机端分支列表被左侧遮挡的问题
- 让移动端分支切换、新建分支与状态提示具备稳定可读性
- 保持桌面端现有分支浮层交互不变
- 尽量复用现有分支状态与行为逻辑，不重做业务流

## 非目标

- 不改分支读取、切换、创建接口
- 不改桌面端分支菜单结构
- 不重做输入器整体布局
- 不引入新的全局状态管理

## 方案对比

### 方案 A：继续使用浮层，只补定位修正

在移动端保留现有 popover 形态，通过 `max-width`、`clamp()`、动态 `left/right` 或 `position: fixed` 把菜单拉回视口内。

优点：

- 改动表面最小
- 桌面端和移动端结构一致

缺点：

- 面板已包含提示区、列表和表单，本质上不再是轻量菜单
- 需要额外处理滚动、软键盘、窗口 resize 和安全区
- 长列表和表单在小屏上仍然拥挤

### 方案 B：推荐方案，桌面保留浮层，移动端改为底部抽屉

桌面端继续使用当前浮层；手机端点击分支 chip 后，改为打开独立的 `Bottom Sheet`。

优点：

- 从交互模型上消除视口越界问题
- 更适合承载“提示 + 列表 + 表单”的复合内容
- 长分支名、长列表和创建表单都有稳定空间
- 与移动端输入器的底部操作区语义一致

缺点：

- 需要为移动端增加一套展示容器
- 需要补遮罩、关闭动作与抽屉样式

### 方案 C：移动端把分支列表拆到独立页面或全屏弹层

点击分支 chip 后进入独立页面，或打开接近全屏的 modal。

优点：

- 空间最充足
- 可继续扩展搜索、最近分支等功能

缺点：

- 交互过重，不适合这轮优化
- 超出“最小直接相关改动”范围

结论：采用方案 B。

## 推荐设计

### 1. 桌面端保留现有浮层

- `> 720px` 继续使用当前 `thread-composer-branch-menu`
- 位置、尺寸和行为保持现状
- 不改变已有分支切换与创建交互

### 2. 移动端切换为 Bottom Sheet

- `<= 720px` 时，点击分支 chip 不再显示相对触发器的窄浮层
- 改为显示覆盖在 composer 之上的底部抽屉
- 抽屉不依赖分支 chip 的横向位置，因此不会再向左越界

推荐尺寸：

- 宽度：`calc(100vw - 24px)`
- 左右边距：12px
- 最大宽度：520px
- 最大高度：`min(70vh, 32rem)`
- 底部补 `env(safe-area-inset-bottom)`

### 3. 抽屉结构分三段

#### 顶部区

- 拖拽把手或轻量分隔条，强化“抽屉”语义
- 标题：`切换或创建分支`
- 副标题：优先显示阻塞摘要；若无阻塞则显示“分支操作会作用于当前工作区”
- 右上角可选关闭按钮

#### 中部内容区

- 显示 dirty summary chips
- 如有 dirty path 预览，只保留前 2 条
- 如有 persisted approval records，只显示前 2 条，其余用“还有 N 条”摘要代替
- 分支列表独立滚动，优先保证至少能看到 3 个分支项

#### 底部操作区

- 固定显示新建分支输入框与“创建并切换”按钮
- 底部区域不参与列表滚动
- 在超窄屏下优先使用上下布局，避免输入框和按钮横向挤压

### 4. 信息密度收敛

移动端不应把所有辅助说明完整铺开，否则首屏会被提示块占满。

因此建议：

- `branchBlockedSummary` 仅显示一段摘要，不堆叠多层说明
- `branchDirtyPreviewPaths` 最多展示 2 条
- `branchPersistedRecords` 最多完整展示 2 条
- `branchGlobalApprovalHint` 仅在存在全局待处理审批时显示
- 当前分支 badge 保留，但分支项正文始终单行截断

### 5. 统一业务状态，分离展示层

移动端抽屉与桌面浮层仅在展示层不同，状态与行为保持复用：

- 继续使用同一个 `isBranchMenuOpen`
- 继续复用 `availableBranches`
- 继续复用 `isBranchLoading`、`isBranchSwitching`
- 继续复用 `branchBlockedReasons`、`branchPersistedRecords`
- 继续复用 `newBranchName`、`onSelectBranch`、`onCreateBranch`

这样可以把这轮改动限制在模板分支与样式层，避免改动业务逻辑。

## 实现落点

- `src/components/content/ThreadComposer.vue`
  - 为分支菜单增加移动端/桌面端双形态渲染
  - 增加移动端底部抽屉、遮罩和关闭入口
  - 补充手机端样式、抽屉布局和安全区处理
- `tests/mobileBranchMenuLayout.test.mjs`
  - 锁定移动端抽屉模式、尺寸约束和桌面端保留浮层的样式信号

如实现过程中发现模板过于臃肿，可再评估是否拆出局部展示组件；本轮优先不拆组件。

## 验收标准

- 手机窄屏打开分支面板后，左侧不再被裁切
- 长分支名不会把当前分支 badge 或主按钮挤出容器
- 分支较多时，中部列表可以滚动，底部新建分支区始终可见
- 存在 dirty/persisted 信息时，首屏仍能看到分支列表入口
- 桌面端分支菜单行为保持不变

## 验证策略

- 先写样式/模板约束测试，确认当前源码尚未具备移动端抽屉形态
- 再做最小实现，确保只影响移动端展示
- 运行 `node --test tests/mobileBranchMenuLayout.test.mjs`
- 运行 `npm run build`
