# 移动端输入器与项目选择样式设计

## 背景

当前手机端有两个直接影响输入体验的问题：

- 聚焦输入框后，iOS Safari 会放大页面，导致输入器底部控件显得被挤压，发送按钮视觉上变形
- 首页项目选择下拉仍按触发器左对齐展开，在手机窄屏下会明显偏左，不在视觉中心

这两个问题都属于移动端样式约束不足，而不是数据流或交互逻辑错误。

## 目标

- 手机端点开输入框时，不再触发浏览器自动放大，发送按钮保持稳定圆形和点击区
- 手机端首页项目选择列表相对内容区域水平居中显示，不贴左边缘
- 桌面端视觉和交互保持不变

## 非目标

- 不改输入器功能、提交逻辑、分支切换逻辑
- 不重做桌面端输入器布局
- 不新增独立移动端组件

## 方案对比

### 方案 A：只补按钮尺寸

只给发送按钮加固定宽高，并减小底部控件间距。

优点：

- 改动最小

缺点：

- 不能解决 iOS 聚焦输入框自动放大的根因
- 也无法改善长分支名对底部布局的挤压

### 方案 B：推荐方案，移动端输入器约束收口

在手机端同时处理三件事：

- 输入框字体提到 16px，避免 iOS focus zoom
- 底部控件区增加 `min-width`/`flex` 约束，让分支选择可压缩、发送按钮固定尺寸
- 首页项目下拉菜单在手机端改为居中展开，并限制最大宽度

优点：

- 能同时解决放大、挤压和下拉偏左三个表现
- 仅改样式层，不改交互结构

缺点：

- 需要同时修改输入器和首页样式

### 方案 C：重排移动端输入器结构

把项目/分支选择移到独立一行，底部只保留模型和发送按钮。

优点：

- 手机端布局最稳定

缺点：

- 交互变化较大，不适合这轮小修

结论：采用方案 B。

## 推荐设计

### 1. 输入框聚焦时阻止 iOS 自动放大

在手机断点下把 `thread-composer-input` 字号提升到 `16px`，保持合适行高，避免 Safari 因输入字体过小而自动缩放页面。

### 2. 底部控件区改成“可压缩选项 + 固定动作”

- `model / reasoning / branch` 仍在同一行
- 发送按钮和停止按钮固定尺寸，不参与压缩
- 分支胶囊允许收缩，文本改为截断
- 状态组允许在剩余空间内压缩，而不是把动作按钮顶出

### 3. 首页项目下拉菜单在手机端居中

- 只在手机端对 `new-thread-folder-dropdown` 生效
- 下拉容器水平居中展开
- 菜单宽度限制为 `min(20rem, calc(100vw - 1.5rem))`
- 保留桌面端现有跟随触发器的表现

## 实现落点

- `src/components/content/ThreadComposer.vue`
  - 修复输入框 focus zoom
  - 收紧手机端底部控件布局约束
- `src/App.vue`
  - 为首页项目下拉补手机端居中样式
- `tests/mobileComposerLayout.test.mjs`
  - 锁住输入器和首页项目下拉的手机端样式约束

## 验证策略

- 先写失败测试，确认当前源码缺少移动端居中/固定尺寸/16px 输入字号约束
- 再做最小样式实现
- 运行 `node --test tests/mobileComposerLayout.test.mjs`
- 运行 `npm run build`
