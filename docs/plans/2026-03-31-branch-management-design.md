# 分支能力设计文档

## 背景与目标

当前项目已经展示当前线程关联的 Git 分支，但并不支持切换或创建分支。UI 上的分支 chip 带有下拉箭头，容易让用户误以为该能力已经存在；而现有架构又是共享 `cwd` 工作区模型，一旦直接增加 `git switch`，影响范围会立刻扩散到同一工作区下的所有线程、diff 视图和后续文件预览逻辑。

本设计文档的目标是：

- 为当前项目补充可控的分支切换与创建能力。
- 明确短期路线与长期路线，避免把共享工作区方案误包装成 Codex app 同等级的 worktree 方案。
- 先给出一条可以落地、风险可控的第一阶段实现路径。

## 现状问题

### 1. 交互承诺与实际能力不一致

当前分支信息仅用于展示，没有配套的查询、切换、创建分支操作。用户在输入框底部看到带箭头的分支 chip，会自然预期可以点击展开菜单，但实际没有行为。这属于明显的交互承诺过量。

### 2. 架构上是共享工作区，不是隔离工作副本

当前项目的 Git 能力都直接建立在 `cwd` 之上：

- workspace diff 基于当前 `cwd`
- 文件预览基于当前 `cwd`
- 线程与项目分组也都围绕 `cwd` 建立

这意味着分支切换是工作区级动作，不是线程私有动作。只要切换了当前工作区分支，同一 `cwd` 下所有线程观察到的仓库状态都会改变。

### 3. 缺乏安全边界

当前项目尚未建立以下阻塞条件判断：

- 工作区是否干净
- 是否存在运行中的 thread / turn
- 是否存在排队消息
- 是否存在待处理审批请求

如果没有这层保护，分支切换很容易引入上下文错乱、工作树污染和用户操作不可逆的风险。

## 路线对比

### 路线 A：第一阶段受限版分支能力（推荐当前实施）

定义：

- 仍然作用于当前工作区
- 支持查询当前分支与本地分支列表
- 支持切换已有本地分支
- 支持基于当前分支创建并切换到新分支
- 只在工作区干净、无线程执行、无排队消息时允许操作

优点：

- 与当前架构兼容，改动集中
- 实现成本可控
- 能较快提供用户可感知的分支能力

缺点：

- 仍然是共享工作区语义
- 同一 `cwd` 下所有线程会一起受到影响
- 无法达到 Codex app 的 worktree 隔离体验

### 路线 B：长期 worktree 路线（推荐作为长期演进）

定义：

- 新线程或新任务不直接复用当前工作区，而是为其创建独立 worktree
- 分支切换、创建分支在 worktree 内完成
- 主工作区只负责接收结果或合并结果

优点：

- 更接近 Codex app 的设计哲学
- 多线程/多任务之间隔离度高
- 分支与线程语义更加一致

缺点：

- 超出当前项目架构边界
- 需要新增 worktree 生命周期管理、映射关系、清理策略与 UI 概念
- 不适合作为第一阶段目标

## 设计结论

当前阶段应优先实施路线 A，并把它明确标记为“共享工作区下的受限版分支能力”，不要误导为线程私有或 worktree 级别的能力。

长期若要对齐 Codex app 的体验与安全边界，应再单独规划路线 B。

## 第一阶段范围

### 支持的能力

- 查看当前工作区当前分支
- 查看当前工作区本地分支列表
- 切换到本地已有分支
- 创建新分支并立即切换

### 明确不做

- 自动 stash
- 强制切换
- 切换远程分支
- detached HEAD 恢复
- merge / rebase 中间态处理
- worktree 创建与回收

## 第一阶段行为约束

以下任一条件成立时，不允许切换或创建并切换分支：

- 当前工作区存在未提交改动
- 当前选中线程存在运行中的 turn
- 当前选中线程存在排队消息

可选增强：

- 如果当前工作区存在待处理审批请求，也一并阻塞

### 语义声明

- 分支切换是工作区级能力，不是线程级能力
- 成功切换后，同一 `cwd` 下所有线程观察到的 Git 状态都会更新
- 历史线程上展示的 `branch` 代表线程摘要记录值，不必强制与当前工作区分支实时完全一致

## 方案设计

### 后端

在 bridge 层补充 Git 分支相关接口：

- `GET /codex-api/git/status?cwd=...`
- `GET /codex-api/git/branches?cwd=...`
- `POST /codex-api/git/branch/switch`
- `POST /codex-api/git/branch/create-and-switch`

返回值至少包含：

- `currentBranch`
- `branches`
- `isRepo`
- `isDirty`
- `blockedReasons`

命令策略：

- 当前分支：`git branch --show-current`
- 本地分支列表：`git branch --list --format='%(refname:short)'`
- 工作区是否干净：`git status --porcelain`
- 切换分支：`git switch <branch>`
- 创建并切换：`git switch -c <branch>`

### 前端状态层

在桌面状态层新增工作区级分支状态：

- 按 `cwd` 维护分支信息
- 维护 `isSwitchingBranch`
- 维护 `branchBlockedReason`

刷新时机：

- 选中线程变化
- 切换成功
- 创建分支成功
- 手动打开分支菜单时

### 前端 UI

将当前分支 chip 从纯展示元素升级为按钮：

- 点击后打开分支菜单
- 菜单显示当前分支、本地分支列表、新建分支输入框
- 若被阻塞，保留菜单但禁用执行按钮，并直接展示阻塞原因

## 风险

### 1. 共享工作区副作用

用户可能误认为操作只影响当前线程。需要在交互文案中明确说明“影响当前工作区”。

### 2. 状态刷新不一致

切换分支后如果线程列表、workspace diff、文件预览没有同步刷新，会出现 UI 与工作区实际状态不一致的问题。

### 3. 阻塞规则不完整

如果只判断 `inProgress`，忽略排队消息或审批请求，仍可能在边界场景下导致操作风险。

## 验收标准

- 当前工作区干净且无线程执行时，可以切换已有本地分支。
- 当前工作区干净且无线程执行时，可以创建新分支并立即切换。
- 任一阻塞条件成立时，UI 明确展示不可操作原因。
- 切换成功后，当前分支展示、线程信息和 workspace diff 能刷新到位。
- 构建验证通过：`npm run build`
